# NexLex — Analytics Plan

> **Status**: Living document. · **Last updated**: 2026-07-16
> Tools, the event schema (versioned here — the code implements THIS file), dashboards, and monitoring. Rule: if a release's Go/No-Go references a number, the event producing it ships **in or before** that release, and is verified end-to-end before users arrive.

---

## Stack (all free tiers; total cost ₹0 until well past V1.0)

| Concern | Tool | Notes |
|---|---|---|
| Product analytics | **PostHog Cloud** (free: 1M events/mo) | Anonymous IDs; cookieless mode on web (no consent banner burden); one project, `platform` property separates web/app |
| Crash reporting | **Sentry** (free tier) | `@sentry/nextjs` + `@sentry/react-native`; release tagging = git tag; source maps via CI |
| Uptime | **UptimeRobot** (free) | `/` + `/api/v1/health`, 5-min interval, email alert |
| SEO | **Google Search Console** | Sitemap already generated by `apps/web`; submit at V1.0 |
| DB health | Supabase dashboard + advisors | Weekly advisor check (standing habit); free-tier daily backups ~7-day window |
| Metrics log | `docs/metrics-log.md` | Monday numbers, hand-written — the act of writing them is the review |

Explicitly not now: data warehouse, session replay (PostHog replay maybe at V1.0 for funnel debugging — decide then), custom event pipeline, BigQuery. Premature.

## Identity

PostHog anon-id (`distinct_id`) per device. No login prompt exists pre-V2.0, so no aliasing complexity. Beta cohort tagging: invite links carry `?c=beta-1` → stored to `localStorage` → sent as `cohort` property on every event (survives navigation; verified before invites go out — a mis-tagged cohort invalidates the beta's learning window).

## Event schema v1 (implement exactly; changes bump the version here)

Naming: `snake_case`, past-tense verbs. Common properties on every event (set once in the client wrapper): `platform` (`web`|`android`), `app_version`, `cohort` (nullable).

| Event | Fires when | Properties | Ships in |
|---|---|---|---|
| `$pageview` | PostHog auto (web) / screen nav (app) | path | V0.1 |
| `act_opened` | act detail viewed | `act` | V0.1 |
| `section_viewed` | section reader rendered | `act`, `number`, `via` (`search`\|`browse`\|`mapping`\|`deeplink`\|`share`\|`recents`\|`bookmark`) | V0.1 |
| `mapping_card_viewed` | mapping card rendered in reader | `mapping_type`, `source_act`, `target_act` | V0.1 |
| `mapping_lookup` | Mapping tab / landing lookup submitted | `parsed_act`, `parsed_section`, `found` (bool) — **never the raw free-text query** (privacy: could contain personal case detail) | V0.1 |
| `search_performed` | FTS search submitted | `query_len`, `results_count` | V0.1 |
| `search_result_clicked` | result tapped | `rank` | V0.1 |
| `landing_lookup_submitted` | hero lookup on `/` | `found` | V0.2 |
| `share_clicked` | share button | `act`, `number`, `channel` (`whatsapp`\|`telegram`\|`copy`\|`native`) | V0.2 |
| `feedback_submitted` | widget submit | `score` (1–5), `has_text` (bool; text goes to DB, not PostHog) | V0.2 |
| `bookmark_added` / `bookmark_removed` | star toggled | `act`, `number` | V0.5 |
| `recents_resumed` | continue-reading tapped | `act`, `number` | V0.5 |
| `fake_door_clicked` | any coming-soon vote | `feature` (`ai_explain`\|`daily_mcq`\|`offline`) | V0.5 |
| `error_boundary_hit` | React error boundary | `route` | V0.5 |
| *(V1.1 events)* | *defined in the G1 addendum before build* | | V1.1 |

**`via` discipline:** `section_viewed.via` is the single most analytically valuable property (it decomposes the product into its entry paths — search vs mapping vs SEO deep links). Every new navigation path added later MUST thread a `via` value or the PR doesn't merge.

## Dashboards (PostHog, built at V0.1, named exactly)

1. **NexLex Core** (the Monday board): WRR (north star), DAU/WAU, activation %, D1/D7/D30 cohort grid, sections/session median, lookups/user/wk.
2. **Funnels**: `$pageview(/)` → `landing_lookup_submitted` → `section_viewed` (message test E-2); `search_performed` → `search_result_clicked` (search quality); `section_viewed` → `mapping_card_viewed` (wedge engagement).
3. **Demand signals**: `fake_door_clicked` by feature (unique users, not clicks); `share_clicked` rate; `via` breakdown of `section_viewed`.
4. **Quality**: Sentry crash-free %, `error_boundary_hit` by route, `mapping_lookup` `found=false` rate (parser gaps), `search_performed` zero-result rate (content/FTS gaps — these two directly feed content backlog).

## Crash reporting & monitoring runbook

- Sentry alert → founder email + the beta WhatsApp group check. P0 = crash on read path or wrong statute text displayed; fix-or-rollback within 24h (rollback steps: [launch-checklist.md](launch-checklist.md) §Rollback).
- **Wrong-content reports are Sev-0** regardless of crash metrics — trust in text accuracy is the moat; a user-visible content error outranks any feature work, always.
- UptimeRobot down-alert: check Vercel status → Supabase status → recent deploy; promote previous deploy first, diagnose second.
- Weekly: Supabase advisors, PostHog event-volume sanity (a silent tracking regression is a lost week of learning — treat instrumentation breakage like a prod outage).

## Privacy posture (matters for Play data-safety form + trust)

Anonymous analytics only; no PII events; no raw search/lookup text in analytics; feedback text stored in Supabase (`feedback` table, RLS: anon INSERT-only) not in PostHog; privacy page states all of this in plain language. When accounts arrive (V2.0), this section gets rewritten *first*.
